# Scientific Scope & Limitations

**TL;DR** — The contribution is the **architecture**: showing that proprietary and open-source models can be compared **dynamically within one pipeline** using isolated parallel tracks and per-case side-by-side results. It is **not** a statistical ML benchmark. Data leakage and the demo-data split are accepted and documented below; the LLM evaluators double as case-level classifiers so proprietary vs. open-source can be compared against a known label.

## In scope (what we demonstrate)

- **One pipeline, many models.** Predict and Evaluate each fan out to multiple models in parallel (`Predictions` / `Evaluations` are 1:many on `Claims`). See [fraud-detection-pipeline.md](ai/fraud-detection-pipeline.md).
- **Isolated tracks.** Each evaluation consumes exactly one prediction (`basedOnPrediction`), so a proprietary lane (RPT-1 → Claude) and an open-source lane (custom ML → gpt-oss/gemma) run independently and are compared side by side.
- **Proprietary vs. open-source, dynamically selectable.** The run-config chooses which models run per case; defaults to the two tracks but supports comparing open-source sizes against each other.
- **Operational comparability.** Each run records `status`, `latencyMs`, and (for LLMs) **output tokens** so cost/latency can be compared alongside accuracy.

## Out of scope (explicitly not claimed)

These are deliberate non-goals and should be stated as such in any writeup or presentation:

- **No full benchmark harness.** No significance testing, confidence intervals, or aggregated accuracy/F1 across a held-out test set as a headline result.
- **No ML-algorithm tuning / data-split rigor.** The custom ML models are used as-is; the train/validation/test split is not the scientific focus.
- **No structure-extraction accuracy study.** The Structure Agent's field-extraction quality is not measured.
- **No fairness / calibration analysis.** No demographic fairness, calibration curves, or bias audit.

## Accepted data-leakage caveat

- **RPT-1 few-shot context** keeps a balanced 50/50 sample from the **training** split as in-context rows (see `fetchContextRows` in `srv/code/on-predictFraud-logic.ts`).
- **Demo/test cases** are drawn from rows that carry a `Tathergang` narrative, and **only the `Training` split has narratives** (`Validation` / `Test` rows have none). So demo cases necessarily come from the training split.
- **Consequence:** there is train/eval overlap. This is an **accepted, documented limitation** — the focus is the architecture, not generalization metrics. The models do not appear to overfit in practice, but no claim of out-of-sample performance is made.
- Demo cases are generated by `scripts/extract-demo-cases.ts` into `test/fixtures/demo-cases.json`, each carrying its `FraudFound_P` as the `actualFraud` label.

## How the comparison is measured

- **LLM-as-classifier.** Each evaluate LLM is prompted to emit a calibrated `fraudProbability` (0–1) and a binary `fraudDecision`, in addition to the narrative `summary` / `riskLevel` / `keyFactors` / `recommendation`.
- **Per-case, against a known label.** For labeled demo cases, each model's `fraudDecision` is compared to the claim's `actualFraud` and surfaced via `decisionCriticality` (3 = correct/green, 1 = wrong/red, 0 = no label) in the Object Page comparison table.
- **Model-vs-model.** Because both lanes run on the same claim, proprietary vs. open-source (and open-source-vs-open-source) can be compared directly per case — decision agreement, probability gap, latency, and output tokens.
- This is a **lightweight, case-level** comparison for demonstration, not a statistical evaluation.

## Minimal reproducibility

- Each evaluation stores its `modelName` and a constant `promptVersion` (`eval-v2-classifier`).
- Evaluation uses a fixed low temperature for stable output; predictions are deterministic given the same context rows.
- No full experiment-config tracking (no seed sweeps, no run manifests) — reproducibility is intentionally minimal and limited to model name + prompt version.
